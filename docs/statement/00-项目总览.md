# 集运订单账单归档系统 - 项目总览

## 一、项目背景

当前系统中，集运订单（yoshop_package）包含完整的物流和费用信息，但缺少对已结算订单的归档管理。需要建立一套账单系统，将已支付的集运订单按周期归档，实现：
- 财务对账的周期性管理
- 已出账订单与待出账订单的数据隔离
- 历史账单的可追溯性

## 二、核心概念

### 2.1 数据状态定义（重要变更）

**旧逻辑（人工发账单时代）**：
- 集运订单的is_pay字段由人工维护，不准确

**新逻辑（系统账单时代）**：
- **待出账状态**：集运订单未生成账单（statement_id IS NULL），is_pay状态不可信
- **已出账未支付**：订单已纳入账单（statement_id有值），但账单未支付（statement.pay_status=1）
- **已出账已支付**：订单已纳入账单，且账单已支付（statement.pay_status=2），此时同步更新订单is_pay=1

**核心原则**：
- 账单是支付状态的唯一真实来源
- 集运订单的is_pay字段由账单支付动作触发更新
- 财务只需管理账单的支付状态，订单状态自动同步

### 2.2 数据关系
```
账单表（Statement）1 ←→ N 集运订单（Package）
一期账单包含多个集运订单
一个集运订单只能属于一期账单
```

## 五、技术要点

### 5.1 事务处理
账单生成必须使用数据库事务，确保数据一致性

### 5.2 并发控制
防止同一订单被重复归档

### 5.3 Excel生成优化
- 大批量数据使用流式写入
- 缓存模板文件
- 异步生成（队列处理）

### 5.4 配置优先级
```
1. 客户专属配置（最高优先级）
   ↓ 如果不存在
2. 历史单价表（导入的TXT/Excel）
   ↓ 如果不存在
3. 全局默认配置
   ↓ 如果不存在
4. 系统兜底（固定单价46元/KG）
```

## 六、业务流程图

```
[待出账订单池（不看is_pay）] 
    ↓ 财务筛选（客户+日期）
[选择订单列表]
    ↓ 勾选 + 点击"生成账单"
[创建账单记录] → [绑定订单] → [计算汇总] → [生成Excel]
    ↓
[账单草稿(pay_status=1)] → 确认 → [已确认账单] → 发送客户 → [已发送账单]
    ↓ 客户付款
[财务点击"标记为已支付"]
    ↓ 触发
[账单pay_status=2] + [所有关联订单is_pay=1同步更新]
    ↓
[订单真实支付状态确认，数据一致性保证]
```

## 九、风险点

1. **数据一致性**：账单和订单状态同步必须使用事务
2. **并发问题**：多人同时生成账单可能导致订单重复归档
3. **Excel性能**：大批量订单生成Excel可能超时
4. **历史数据**：旧订单的is_pay字段不准确，需要兼容处理
5. **计价复杂度**：自定义公式需要安全评估，防止注入攻击
