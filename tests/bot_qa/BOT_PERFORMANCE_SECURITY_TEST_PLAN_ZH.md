# Bot 性能与安全测试计划

基于技术说明文档，本文件定义了 Bot 系统在性能 (Performance)、基础架构异常处理 (Exception Handling) 与安全性 (Security) 三方面的非功能性测试计划。

---

## 1. 性能测试 (Performance Testing)

| TC ID | 场景与技术要求 | 工具执行步骤 | 期待结果验收 (SLA) | 状态 |
|---|---|---|---|---|
| TC_PERF_01 | **处理大并发消息**<br>模拟 100 位用户同时交互。 | 使用 JMeter 或 K6 向接收 Webhook 的 API 并发打入 100 条消息。 | - 0% 错误率，未漏消息。<br>- 响应时间 (Response time) < 2 秒。 | [ ] |
| TC_PERF_02 | **Redis 缓存命中率检查**<br>降低数据库压力。 | 脚本在一分钟内发送 1000 个查询配置和简单信息的请求。通过 Redis Monitor/Grafana 查看监控。 | - 缓存命中率 (Cache Hit) 达到 80% - 90%。<br>- 大幅减少直接查 MySQL 的消耗。 | [ ] |
| TC_PERF_03 | **数据库连接池承载**<br>防止 MySQL 会话过载。 | 直接用压测脚本向”产生/插入订单“ API 打入 200 个并发请求。 | - MySQL 没有报 "Too many connections" 的连接溢出错误。<br>- 队列排队消化良好。 | [ ] |
| TC_PERF_04 | **旧 Webhook 积压消化处理**<br>系统宕机恢复后涌入的数据。 | 1. 停止 Bot Worker 消费队列。<br>2. 堆积发送 500 条消息进 FB。<br>3. 重新开启服务。 | - 系统顺理成章将 500 条消息载入队列。<br>- 按序并且无卡顿不丢弃地消化完毕。 | [ ] |

## 2. 基础设施异常场景 (Infrastructure Exception Handling)

| TC ID | 场景与技术要求 | 工具执行步骤 | 期待结果验收 (SLA) | 状态 |
|---|---|---|---|---|
| TC_EXC_01 | **Meta 官方 API 请求超时**<br>服务器连 Facebook/Instagram 网络延迟中断。 | 防火墙直接屏蔽前往 Meta 平台的 IP 报文或模拟响应极其缓慢 (>10s)。 | - Bot 记录超时错误并将待发消息挪入 重试队列 (Retry Queue)。<br>- 其他用户的进程一切照常进行，绝无资源被强行占死的情况。 | [ ] |
| TC_EXC_02 | **第三方物流 API 宕机备用方案**<br>交货物流查询无法响应。 | 进行物流跟踪请求时，将 API 返回设为 500 内部错误。 | - Bot 启动降级逻辑 (Fallback)。<br>- 返回友好文字：“物流第三方网络维护中，请稍后刷新重试”。 | [ ] |
| TC_EXC_03 | **数据库断开连接后自动自愈**<br>防止雪崩引发连锁故障。 | 1. 终止 MySQL 服务进程。<br>2. 测试发消息给 Bot。<br>3. 恢复 MySQL 后再次发。 | - 终止时: Bot 礼貌回复“系统繁忙中”。<br>- 开启时: 立刻重新建立连接，读写毫无延迟介入。 | [ ] |
| TC_EXC_04 | **Redis 断开连接处理**<br>缓存组件脱机。 | 1. 终止 Redis 服务。<br>2. 请求获取通常存入缓存里的配置值。 | - 触发 Fallback，自动由底层向 MySQL 获取。<br>- 日志管理生成 High (高级别) 错误警告。 | [ ] |

## 3. 安全性测试 (Security Testing)

| TC ID | 场景与技术要求 | 工具执行步骤 | 期待结果验收 (SLA) | 状态 |
|---|---|---|---|---|
| TC_SEC_01 | **绕过 API Key 伪造**<br>防止未授权程序进行端点访问。 | 使用 Postman 测试创建订单节点，不加 API Key 标头或者是乱填一把乱码。 | - 中间件立刻回绝拦截。<br>- 抛出 HTTP 401 Unauthorized 的错误码。 | [ ] |
| TC_SEC_02 | **防范 SQL 注入黑客攻击**<br>维护核心表与数据安全。 | 在“单号”这种接收前端输入的框里注入带有威胁性的语句：`' OR 1=1; DROP TABLE users;--`。 | - 恶意字符串接受净化过滤 (Sanitized)。<br>- 解析防范报错 400 Bad Request。数据库不会遭到破坏。 | [ ] |
| TC_SEC_03 | **篡改 wxapp_id 越权查询**<br>防范同行调取商户订单和顾客资料。 | 已登录账号 `wxapp_id = 1`，抓包并将参数字段强制篡改为 `wxapp_id = 2` 后发包。 | - 请求内绑定参数如果和派发 Token 的所属字段对不上，即刻拦截。<br>- 返回 HTTP 403 Forbidden 封禁访问页。 | [ ] |
| TC_SEC_04 | **并发访问限流 (Rate Limit)**<br>阻挡网络机器人 DDOS 暴力攻击。 | 配置测试攻击脚本：每 60 秒对某查单 API 拼命打入 110 条请求。 | - 服务器准确限制在容许次数 (如100)。<br>- 一旦到达第101次立马抛出 HTTP 429 Too Many Requests 阻挡访问。 | [ ] |
